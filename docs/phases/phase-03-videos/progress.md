### SI-03.8 — Processor do job video.process
- **Status:** completed
- **Tests:** 6 passing (5 unit + 1 integration)
- **Observations:**
  - Confirmed via context7 that `@nestjs/bullmq` v11 uses `@Processor(queue) class X extends WorkerHost { process(job) }`, not the older `@nestjs/bull` `@Process()` method-decorator style — this determined the processor's shape.
  - Critical architecture finding: `@nestjs/bullmq`'s discovery explorer scans the *entire* DI graph — any `@Processor`-decorated provider instantiated anywhere in a running Nest app starts a live BullMQ `Worker` consuming that queue, regardless of which "logical module" declared it. Since `VideoProcessingModule` (queue registration + `FFmpegService`) is imported by both `AppModule` (via `VideosModule`, the producer side in `nestjs-api`) and `WorkerModule` (the dedicated worker process), adding `VideoProcessingProcessor` directly to `VideoProcessingModule` would have made `nestjs-api` *also* consume `video.process` jobs — racing the dedicated worker and crashing on every job it grabbed, since `nestjs-api`'s image has no `ffmpeg`/`ffprobe` binaries (SI-03.7 precedent). Fixed by creating a new `VideoProcessingWorkerModule` (`src/video-processing/video-processing-worker.module.ts`) that imports `VideoProcessingModule` + `TypeOrmModule.forFeature([Video])` + `StorageModule` and declares `VideoProcessingProcessor` as its own provider; `WorkerModule` now imports this new module instead of `VideoProcessingModule` directly. `VideoProcessingModule` itself is untouched, so `nestjs-api` never instantiates the processor.
  - `WorkerModule` had no database connection at all before this SI (SI-03.6 only wired Redis) — added `TypeOrmModule.forRootAsync` (mirroring `AppModule`'s block) plus `databaseConfig`/`storageConfig` to its `ConfigModule.load` array, since the processor needs a real `Video` repository to persist status transitions. Added `db` and `minio` to `video-worker`'s `depends_on` in `compose.yaml` to match (previously only depended on `redis`, per SI-03.6's precedent of not adding dependencies before the code actually needed them).
  - `StorageService` (SI-03.3) had no method to fetch/write raw object bytes — only multipart-upload and presigned-URL methods, since prior SIs only needed client-side direct-to-storage transfer. Added `getObjectStream(key)` (returns `GetObjectCommand`'s `Body` as a `Readable`) and `putObject(key, buffer, contentType)` (wraps `PutObjectCommand`, used for the generated thumbnail).
